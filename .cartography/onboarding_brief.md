# FDE Day-One Onboarding Brief

Q1: What is the primary data ingestion path?  
The primary data ingestion path involves the `edxorg_archive.py` module, which processes tracking logs and updates them. This is evident from the frequent commits in `edxorg_archive.py` (10 commits) and its role in handling `edxorg_raw_data_archive` and `edxorg_raw_tracking_logs`. Additionally, the `dbt-local-dev.py` script centralizes ingestion by managing metadata in `_glue_source_registry` and integrating with dbt staging models.  

Q2: What are the 3-5 most critical output datasets/endpoints?  
1. `_glue_source_registry` (central metadata table used across multiple modules).  
2. `edxorg_raw_data_archive` and `edxorg_raw_tracking_logs` (processed by `edxorg_archive.py`).  
3. B2B organization exports (CSV reports from `b2b_organization` module).  
4. Reconciled partitions for course data (from `dg_deployments\reconcile_edxorg_partitions.py`).  
5. Staging models generated by `dbt-create-staging-models.py`.  

Q3: What is the blast radius if the most critical module changes its interface?  
The most critical module is `bin\dbt-local-dev.py`, which orchestrates dbt operations and metadata management. Changes to its interface would disrupt:  
- Dbt staging model generation (`bin\dbt-create-staging-models.py`).  
- Local environment setup (`_glue_source_registry` metadata).  
- Other modules relying on its SQL commands (e.g., `SELECT` queries on `_glue_source_registry`).  
Blast radius includes all dbt-related workflows and metadata-dependent processes.  

Q4: Where is the business logic concentrated vs distributed?  
- **Concentrated**: Business logic is in domain-specific modules like `b2b_organization` (exports, partitions, sensors) and `edxorg_archive.py` (tracking log processing).  
- **Distributed**: Infrastructure and utility logic is in `bin\dbt-local-dev.py` (local dbt setup) and `ol-orchestrate-lib` packages (e.g., `glue_helper.py` for AWS Glue operations).
